Pre-redesign audit.

A read on what the app looks like today, where the visual language gets in the way of the product, and what to lock down before generating another round of dashboard directions. The aesthetic of this document is also the recommended destination — restrained, calm, single-accent, no neon.

TL;DR

read first

The one thing

Your engineering is production-grade — 359 source files, 420 tests, real analysis logic across leak detection, range compliance, ICM, final-table, villain classification. Your visual layer is prototype-grade — OLED black, neon green, glassmorphism, mono-everywhere, every metric rendered with the same weight.

The gap between those two things is the entire problem. A buyer evaluating this product can't tell from the UI that the analysis underneath is serious. That's a fixable framing problem — not a code problem.

Three commitments before any redesign: pick one product name, kill the casino-HUD identity, and impose a single confidence/severity system across every page. After that, "Tournament Lab" as the visual direction is the right bet.

01

Brand & positioning

the identity problem

The app currently has at least five competing identities, none of them committed to.

  • "Poker Analyzer HUD" — browser tab title and what most surfaces say
  • "Poker Analyzer / App Settings" — sidebar header (neon-green mono, reads as a config menu, not a brand)
  • "EV Platform" — sidebar footer v0.2.0 — EV Platform
  • "CareerScope" — Career page section title, name borrowed from SharkScope
  • "Tournament Lab" — your stated design direction, but not used anywhere in-product yet
  • "The Arena" / "Fault Fixer" — sub-feature names that read like game modes from a different app

This proliferation is the loudest signal of "agent-built prototype". A serious product has one name on the door and one POV. Right now a buyer can't tell what to call the thing.

The brand mark itself

Issues: the spade-emoji + monospace + neon-green combo reads as a poker-themed admin panel, not as analytics. "App Settings" as a subtitle suggests the whole sidebar is settings. The footer adds "EV Platform v0.2.0" which is internal vocabulary.
Proposed
Tournament Lab
private analysis
Why: dedicated geometric mark (not an emoji), single product name, sans-serif wordmark, a quiet descriptor under it. No version string in the chrome — version belongs in Settings → About, not in the navigation. Reads as a tool, not a HUD.

Recommendation: commit to a single product name and stop using it as decoration. "Tournament Lab" is the one I'd pick from your existing vocabulary — it's specific, it's the one name that doesn't sound borrowed, and it leaves room for a poker-adjacent product (cash games, mixed) later without renaming.

02

Color & type system

src/index.css

Your src/index.css declares the theme as "Premium Dark Poker HUD" and ships with #00e676 neon green and #ff3366 neon pink. That string in the source is the honest tell — it's a HUD theme, not an analytics theme.

What goes wrong, specifically

  • Neon green as chrome. --color-accent: #00e676 is used for the brand title, active sidebar item, "load demo" button, status badges, "compliant" rows, success deltas, AND the chart line. When everything important is neon green, nothing stands out.
  • Neon pink for routine deviations. #ff3366 on every off-target stat means a 99.7% VPIP (catastrophic) and an 11.0 AF (slightly high) get the exact same visual weight. The user can't triage by glancing.
  • Glassmorphism on data-dense surfaces. backdrop-filter: blur(16px) on every card backing makes tables and stat grids feel hazy on top of the radial gradient. Beautiful in screenshots, hostile during a 2-hour study session.
  • Mono everywhere. Headings like "HOLD CURRENT STAKE", "STRATEGIC AUDIT", "PLAYER CAREER" are all JetBrains Mono uppercase. Mono should be reserved for IDs, hand keys, amounts, ratios. Body and headings are sans.
  • Suit colors are sensible. Spade-white, heart-red, diamond-blue, club-green is readable and consistent. Keep these — they belong inside the hand-replay card and nowhere else.

Recommended type

Sans · Inter

Keep Inter — it's the right choice. Drop the 800-weight import; you only need 400/500/600.

  • Page title — 28px / 600 / -0.02em
  • Section title — 16-18px / 600
  • Card title — 13-14px / 600
  • Body — 13-14px / 400-500
  • Overline / label — 11px / 500 / 0.1em / uppercase

Mono · JetBrains Mono

Keep — but restrict it to:

  • Numeric metric values (tabular)
  • Hand IDs, player IDs, scenario codes
  • Tiny "10-uppercase-letterspaced" overline labels
  • Keyboard shortcut chips

Never on page titles or headings.

Specific copy issues

  • "PRIVATE SHARKSCOPE" on the Career page directly invokes a third-party product. Per your own IP audit doc, this needs to go.
  • "Competitor-inspired study queue" as a section header. "Competitor-inspired" is a meta-product statement — buyers don't need to know what you were inspired by.
  • "Coach or buyer" copy. "Playable, but blocked from 250 tournaments and 10716 hands. This is the first thing a coach or buyer should understand before opening any chart." — the user is the coach. They're not a buyer in this context. Cut "or buyer".

Cross-cutting issues

apply everywhere

One severity system, applied everywhere

Today the same red is used for "you're playing 99.7% of hands" (catastrophic) and "your AF is 11.0 vs target 2-3" (high but not catastrophic). Lock down four severity bands and use them globally:

  • critical — Z-score > 3 from target, sample > 1000. Use red border-dot only, never red card chrome.
  • high — Z-score 2-3, or Z > 3 with sample 200-1000.
  • medium — Z 1-2, real but not urgent.
  • on-target — within band. Default to neutral foreground, not green.

Sample / confidence is a first-class citizen

Every metric needs to say how confident alongside what. A 99.7% VPIP from 10,716 hands is iron-clad; the same number from 60 hands is noise. Today the app shows BB/100 of -475.2 on 9 hands as if it matters. Three rules:

  • Hand-level stats < 200 hands: render in var(--fg-3) + "low confidence" chip
  • Hand-level stats 200-1000: full opacity, no chip
  • Hand-level stats > 1000: full opacity + small "n=10,716" annotation
  • BB/100 below threshold: do not render at all

Number formatting

Pick one locale and stick to it. Today there's +$361.85 AND $3,4 AND +$18,5 on the same page — that's a mix of US format and European format. Use Intl.NumberFormat('en-US') with two decimal places for currency, and put currency right-aligned in tables.

Loading and empty states

Right now the Villains page shows 44 zero-rows when the data isn't populated, the Profit Heatmap shows a vertical green column, and the Statistics top-10 shows duplicate hands. A "no data yet" empty state is more credible than a broken-looking populated one. Build three reusable shells:

  • Skeleton — used while parsing/computing
  • Empty — used when the user has 0 hands for this view
  • Low-confidence — used when the user has some, but not enough

Animation budget

Today, button:active { transform: scale(0.97) } + scale-105 on active nav + animate-in slide-up zoom-in fade-in entrances on cards. Pick one transition primitive (probably opacity + 4px translate on mount), and one hover state (slight background lift). Drop the active-scale.
